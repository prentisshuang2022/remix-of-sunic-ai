## 1. 角色定义

| 角色代码 | 角色名称 | 描述 | 上下级关系 |
|----------|----------|------|------------|
| `candidate` | 候选人 | 外部用户，只能查看自己的投递记录和面试安排 | 无 |
| `employee` | 普通员工 | 在职员工，访问自己的数据 + AI 助手 | 向 manager 汇报 |
| `manager` | 部门主管 | 自己 + 直属下属的数据，可审批下属请假/考勤 | 管理 employee |
| `hr` | HR 人员 | 全公司员工数据 + 招聘 + 配置 + 培训管理 | 跨部门横向 |
| `interviewer` | 面试官 | 查看自己参与的面试和候选人资料 | 临时角色 |
| `admin` | 系统管理员 | 所有权限，系统配置，角色分配 | 最高权限 |

> **说明**：一个用户可同时拥有多个角色（如 manager + interviewer）。权限取并集。

## 11. 数据可见性规则总结

### candidate（候选人）
- 只能通过候选人门户（如有）查看自己投递的岗位、面试安排、Offer 状态
- 不可使用任何 AI 功能
- 不可查看任何员工信息

### employee（普通员工）
- **自己的**：完整档案（敏感字段脱敏展示）、考勤记录、绩效表单、OKR、培训成绩
- **他人的**：仅可见姓名、部门、岗位、工号（公开字段）
- **AI 助手**：只能基于自己身份提问，AI 不返回他人数据
- 不可访问：招聘模块、他人考勤/绩效、系统管理

### manager（部门主管）
- 拥有 employee 全部权限
- **额外可见**：直属下属的完整档案（不含银行卡等敏感字段）、考勤、绩效、OKR
- **可操作**：审批下属补卡/请假、评分下属绩效、核销下属加班
- **AI 洞察**：可查询下属团队的汇总数据
- 不可访问：非下属员工的数据、招聘模块

### hr（HR 人员）
- **全公司**：所有员工的完整档案（含敏感字段完整显示）
- **可操作**：全部员工管理操作、招聘全流程、培训管理、考勤管理、绩效管理
- **AI 能力**：所有 Agent 场景，查询范围为全公司
- 不可访问：系统管理、角色/权限配置

### interviewer（面试官）
- 基础权限同 employee
- **额外可见**：自己参与面试的候选人资料、面试安排
- **可操作**：提交面试反馈
- **AI 能力**：interview_helper 场景，仅限自己面试的候选人
- 该角色通常与 employee/manager 角色叠加

### admin（系统管理员）
- 所有权限，包括 hr 全部权限 + 系统管理 + 审计日志 + 角色配置 + AI Agent 配置
- 可查看他人 AI 对话（仅审计目的）

## 12. RLS 实现建议

后端实现时建议在数据库层使用 Row Level Security (RLS)：

```sql
-- 示例：员工表 RLS 策略
-- employee 只能看自己
CREATE POLICY employee_self ON employees
  FOR SELECT TO authenticated
  USING (id = current_user_employee_id());

-- manager 可看下属
CREATE POLICY manager_subordinates ON employees
  FOR SELECT TO authenticated
  USING (
    direct_manager_id = current_user_employee_id()
    OR id = current_user_employee_id()
  );

-- hr 可看全部
CREATE POLICY hr_all ON employees
  FOR SELECT TO authenticated
  USING (current_user_has_role('hr'));
```

> 具体实现取决于后端技术栈，以上为 PostgreSQL RLS 参考。应用层也需要做权限校验（Defense in Depth）。
